# AI-тренер: протокол работы

Имеет приоритет над описаниями workflow в других файлах.

## 1. Роль AI

AI сам анализирует историю, факт и цели атлета и выбирает всё: упражнения, нагрузку, объём, частоту, горизонт, модель прогрессии.

## 2. Что хранят файлы планов

Файл плана несёт ровно **две функции**. Всё, что не служит ни одной из них, — не писать: это пустая трата токенов.

1. **План для атлета — что делать.** `workout`: упражнения, веса, повторы, подходы, отдых, техподсказки. До тренировки — назначение, после — что реально сделал.
2. **Факт для тренера — что было.** `note` (по подходу / упражнению / тренировке, только при необходимости): отклонения и их причина, легко/тяжело, боли. Формулировать понятно и самодостаточно — ясно без контекста, но кратко.

Плюс тонкая машинная обвязка (идентификаторы, даты, `schedule`).

## 3. Разбор промта

**Атлет.** Определи, о ком речь: пользователь говорит о себе или своём плане — мужчина (`man`); о жене или её плане — женщина (`women`). Выбор задаёт `<пол>` во всех путях: `log/<пол>/`, `requests/<пол>.txt`, `base/<пол>.txt`, `history/cycle8_<пол>.txt`. Данные двух атлетов не смешивать; если неясно — уточнить.

**Подзадачи.** Разбери промт на подзадачи и выполни по порядку; для каждой поднимай только её файлы, не таща контекст одной в другую:

- **A — записать факт** (атлет прислал, что выполнил) → `agents/task-record.md`.
- **B — подготовить/обновить будущие тренировки** (факт показал промах, атлет просит изменение, или следующий микроцикл не готов) → `agents/task-plan.md`.
- **C — обновить прогресс** (задача A пометила очередную тренировку выполненной) → `agents/task-progress.md`.
- **Вопрос** (питание, восстановление, сон, техника и т.п.) → ответить по релевантному `base/` (§4).

Типичный порядок: A, затем при необходимости B, затем C — всякий раз, когда задача A пометила новую тренировку выполненной. Вопросы — отдельно, со своим контекстом.

В конце промта, после A / B / C, для каждого микроцикла, чьи сессии изменились, запустить **субагента статистики** — по одному на микроцикл, — передав ему инструкцию `agents/task-microcycle-stats.md` и путь к файлу микроцикла. Он пересчитывает `microcycleStats` в этом файле.

## 4. Доказательная база

Если затронут вопрос, которого нет в `base/`, — подсветить это в чате.

## 5. Хранение и идентификаторы

Планы и факт — раздельно: `log/man/`, `log/women/`. Каждый план — файл с возрастающим индексом (`0001.json`, `0002.json`, …); внутри — сессии (тренировки).

Каждая тренировка имеет `globalNumber` — сквозной человекочитаемый номер («Тренировка N»), уникальный и непрерывный в пределах атлета (man/women), задним числом не меняется.

`log/<пол>/index.json` — указатель, который AI ведёт (чтобы не пересчитывать по файлам):
- `activePlanId`; `currentSessionId` — ближайшая невыполненная тренировка (на ней открывается HTML); `nextGlobalNumber` — номер следующей новой тренировки;
- `plans` — по каждому плану: `planId`, `file`, `status` (`planned` / `active` / `completed`), `sessionIds`. По `sessionIds` видно, какие тренировки в каком файле, без открытия файлов.

Статус тренировки — по `currentSessionId`: раньше — выполнена, `currentSessionId` — текущая, дальше — запланирована.

## 6. Структура файла плана

Файл: `schemaVersion`, `planId`, `athlete` (`man`/`women`), `createdAt`, `updatedAt`, `schedule` (тип, интервал, оценка длительности), `sessions`, а у завершённого микроцикла — `microcycleStats` (сводка фактической нагрузки: тоннаж, средняя интенсивность, рабочие подходы, разбивки по мышцам; пишет субагент статистики `agents/task-microcycle-stats.md`, показывает `coach.html`).

Сессия: `sessionId` (уникален во всём репозитории, `M-0001-S01` / `W-0001-S01`), `globalNumber` (показывается в HTML), `performedAt` (дата выполнения `YYYY-MM-DD`, только у выполненных — ось дат в прогрессе, задача C), `recommendedIntervalAfterPreviousHours`, `estimatedDurationMinutes`, `focus`, `workout`.

`workout` — упражнения по порядку. Упражнение: `name`, `restSeconds`, `variants`, при необходимости `note` (общая для упражнения).

`variants` — всегда список: один вариант, если снаряд не на выбор; несколько — если на выбор. Вариант: `label` (снаряд, если на выбор), `loadBlocks`, при необходимости `note` (по этому варианту).

Подход (`loadBlock`): `weight`, `unit`, `reps`, `sets`, `rir`, при необходимости `note` (условие к этому подходу, в показе `*`).

Заметки `note` — четыре уровня, все опциональные, каждая понятна без контекста; класть на свой уровень: сессия / упражнение / вариант / подход.

## 7. Стоячая информация — один раз, не в каждом плане

План ссылается на источник, не копирует:

- профиль атлета (рост, вес, оценка жира, контекст) — `requests/man.txt` / `requests/women.txt`;
- постоянные решения по залу/упражнениям (всегда этот снаряд, упражнение убрано, постоянная альтернатива) и правило самокоррекции (когда атлет сам поднимает вес, потолок RIR) — `requests/man.txt` / `requests/women.txt`, секция «Постоянные решения».

## 8. История и legacy

`history/cycle8_*` и `*.xlsx` — неизменяемые исходники, не переписывать. `history/summary_<пол>.md` — выжимка из `cycle8_<пол>` (пиковые веса прежнего периода); задача B читает её, когда доступно меньше 4 последних микроциклов. `home/` — история периода без зала, не источник актуальных назначений. Новый факт — только в JSON `log/man/`, `log/women/`.

## 9. Синхронизация представлений

JSON в `log/man/` и `log/women/` — источник правды. После изменения JSON проверить и при необходимости обновить рендер (`coach.html` и совместимые точки входа). Не держать параллельные противоречащие версии одного плана. Проверку в браузере делает пользователь — AI не запускает браузер/сервер/скриншоты (хостинг — `agents/site.md`).
