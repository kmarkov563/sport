# Задача C — собрать данные прогресса

Запускается после того, как обновлён факт (задача A) и, при необходимости, план (задача B): берёт свежезаписанные тренировки и дополняет ими сводку весов для графика прогресса.

Атлеты раздельно, ничего не смешивать: у мужчины и женщины свой отчёт и свой запуск.

- **Вход:** `log/<пол>/index.json`, файлы планов `log/<пол>/*.json`, текущий `log/<пол>/progress_report.json` (если есть).
- **Выход:** `log/<пол>/progress_report.json`.

## 1. Что описывает отчёт

Одна серия = одно упражнение в одной колее снаряда. По каждой тренировке в серии лежит одна точка: реальный максимальный рабочий вес того дня и расчётный разовый максимум.

```json
{
  "athlete": "man",
  "lastSession": "M-0003-S01",
  "exercises": [
    {
      "id": "bench_press",
      "name": "Жим лёжа",
      "aliases": ["Жим штанги лёжа"],
      "unit": "kg",
      "color": "#4E79A7",
      "data": [
        { "sessionId": "M-0001-S01", "globalNumber": 1, "date": "2026-07-05", "max": 85, "onepr": 99 }
      ]
    }
  ]
}
```

- `unit` — единица серии: `kg`, `kg_per_hand` (на одну руку) или `kg_per_side` (на одну сторону). Серия однородна по единице; вес в `kg` и вес «на руку» — разные серии.
- `max` — самый тяжёлый рабочий подход дня в единице серии (сплошная линия на графике).
- `onepr` — расчётный разовый максимум (штриховая линия); `null`, если в тот день его посчитать нельзя.

## 2. Что на график не идёт

- Упражнения без измеримого веса: только вес тела (пресс, планка, гиперэкстензия) и «базовый вес тренажёра» без блинов (отведение бедра/ноги на пустом стеке). Если у упражнения веса нет **никогда** — серии для него не создавать.
- Разминочные и подводящие подходы: блок без `weight`, с `weight: 0`, с `rir: "разминка"` или с пометкой про разминку/подводящий. В расчёте `max`/`onepr` не участвуют.
- Пропущенная тренировка (`variants: []`, «не выполнено») — точки нет.

## 3. Вес и разовый максимум

По каждому упражнению берётся выполненный вариант (его `loadBlocks`), из него — только рабочие подходы (см. §2).

**Единицы → в единицу серии.** `kg` как есть. `plates` (плитки) → ×5 кг. `kg_per_hand` и `kg_per_side` не пересчитывать — это и есть единица серии.

- `max` = наибольший `weight` среди рабочих подходов.
- `onepr` = максимум по рабочим подходам от **weight × (1 + (reps + rir) / 30)**, округлить до 1 кг. Считать только по подходам с `reps ≤ 12`; если ни одного такого нет — `onepr: null`. `rir` отсутствует — считать за 0.

`max` и `onepr` могут прийти из разных подходов (тяжёлый сингл даёт `max`, многоповторный — лучший `onepr`) — это нормально. Запас (`rir`) в формуле нужен, чтобы подход не в отказ не занижал оценку.

## 4. К какой серии отнести упражнение

Идентичность серии — пара «название + `label` варианта» (снаряд на выбор): `Румынская тяга`/`Штанга` и `Румынская тяга`/`Смит` — разные серии (веса не переносятся между снарядами). Без `label` — по одному названию.

1. Сопоставить пару со списком `aliases` существующих серий. Совпало — дописать точку туда.
2. **Не совпало — это новое упражнение. Молча не угадывать.** Предложить наиболее логичный вариант (существующую серию того же движения/мышцы либо новую серию) и **спросить пользователя**, куда отнести. Только после ответа: добавить строку в `aliases` этой серии или завести новую серию, затем дописать точку.

Так гранулярность (обычный жим против паузного, штанга против Смита) решает пользователь один раз при первом появлении названия, а не задаётся заранее.

## 5. Цвет

У серии постоянный цвет — задаётся при создании и больше не меняется. Новой серии: `hsl((n × 137.5) mod 360, 65%, 50%)`, где `n` — число уже существующих серий; сохранить как hex. Золотой угол разводит соседние серии по тону.

## 6. Порядок серий

`exercises` — в порядке первого появления упражнения (по `globalNumber`). Страница по умолчанию показывает первые 3; порядок = приоритет. Пользователь может переставить вручную.

## 7. Даты (`performedAt`)

`date` точки = дата из `performedAt` сессии (только число, `YYYY-MM-DD`).

Если у выполненной сессии `performedAt` нет (старые записи), сначала проставить его в файл плана — один раз, задним числом:

- спросить у пользователя дату последней выполненной тренировки этого атлета;
- назад по порядку: внутри одного микроцикла (плана) — через день (−2 дня на сессию), между микроциклами — −3 дня (2 дня отдыха на стыке планов);
- планируемые сессии (от `currentSessionId` дальше) `performedAt` не получают.

Дальше отчёт просто читает `performedAt`.

## 8. Сборка

1. По `index.json` определить выполненные сессии (всё до `currentSessionId`).
2. Если `lastSession` уже равен последней выполненной сессии и старые тренировки не правились — делать нечего.
3. Иначе пересобрать `data` заново по всем выполненным сессиям в порядке `globalNumber`, **сохранив** у существующих серий `id`, `name`, `aliases`, `unit`, `color` (меняется только `data`) — так цвета и разбивка не скачут, а поздние правки прошлых тренировок подхватываются.
4. `lastSession` = последняя выполненная сессия. Записать `log/<пол>/progress_report.json`.

## 9. Страница

`progress_report.json` читает страница прогресса (отдельно man/women): вертикаль — вес, горизонталь — дата, сплошная линия `max`, штриховая `onepr`, цвет — из `color`. Скрытые серии пользователь хранит в кеше браузера. После обновления JSON больше ничего не требуется.
